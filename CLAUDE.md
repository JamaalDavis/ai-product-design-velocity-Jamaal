## PRD Format — Standard Section Order

Every PRD produced in this project follows this 9-section structure. Do not skip or reorder sections.

1. **Problem Statement** — user perspective only. No business metrics here.
2. **Business Context** — connects user problem to company metrics (churn, conversion, ARR). Table: current / target / timeline. Modelled impact from the EIM Impact section.
3. **Hypothesis** — We believe / Because / We'll know it worked when. Falsifiable.
4. **Success Metrics** — primary (leading), secondary (lagging), guardrail. Each with baseline and target.
5. **Scope** — in scope, out of scope, assumptions and dependencies.
6. **User Stories** — 3–6 stories, each with testable acceptance criteria.
7. **Functional Requirements** — numbered (FR-01, FR-02…), grouped by area, each with Must / Should / Could priority.
8. **Analytics Instrumentation** — table: event name, when it fires, additional properties.
9. **Open Questions** — numbered list. Each with owner and default assumption.

Business Context (section 2) is always separate from Problem Statement (section 1). Never merge them.

## Skills

Skills are installed globally at `~/.claude/skills/`. The copies in `skills/` are the source — edit here and reinstall to update.

**Installed skills and what they do:**

| Skill | What it does |
|---|---|
| `/eim-context` | Checks data readiness, synthesises product context into `[company]-context-base.md` |
| `/eim-product-analyst` | Analyses CSVs, produces 3–5 EIM hypotheses ranked by confidence × impact |
| `/hypothesis-to-prd` | Turns one EIM hypothesis into a PRD and/or prototype brief |
| `/build-html-prototype` | Builds a self-contained clickable HTML file from a prototype brief |
| `/figma-make-spec` | Reads a Figma file via MCP, extracts design tokens and components into `[company]-figma-spec.md` |
| `/figma-prototype-builder` | Builds screens and wired navigation directly inside a Figma file using MCP |
| `/ready-to-pitch` | Turns product work, PRDs, prototypes, and research into a slide-by-slide presentation spec with speaker script |

**Skill invocation order:**
```
/eim-context → /eim-product-analyst → PAUSE → /hypothesis-to-prd → PAUSE → /build-html-prototype
                                                                           → /figma-make-spec → /figma-prototype-builder
                                                                           → Figma Make (paste build prompt from prototype brief)
```

The PAUSEs are human judgment checkpoints. Skills handle process. The decision about which signals matter, which hypothesis to pursue, and which solution direction to take — that belongs to the person, not the skill.

## Workflow Conventions

**VS Code file tree check** — after every file generation, check the VS Code sidebar to confirm the file landed in the right folder. Five seconds. Non-negotiable habit in this course.

**Spec drives everything** — when build output and expectations disagree, the first question is always: does the PRD need updating, or does the build need correcting? Never adjust output without checking the spec first.

**Prototype brief is source of truth for prototypes** — when a Figma Make or HTML prototype doesn't match expectations, fix the build prompt in the brief, not the prototype directly.

**Context base before analysis** — always run `/eim-context` before `/eim-product-analyst` on a new company. The context base is what makes hypotheses product-relevant rather than statistically correct but operationally useless.

## Rules

- **Do not modify files in `finwise/data/`** — these are source datasets. Run `generate_data.py` for a fresh copy.
- **Do not modify skill files in `skills/` and expect them to take effect** — Claude Code runs the installed copies at `~/.claude/skills/`. Edit source here, then reinstall.
- **Do not save outputs to the repo root** — everything goes into the appropriate subfolder.
- **Ask before making changes that affect multiple files at once.**
- **Course Docs are planning and reference documents** — do not auto-update them based on inferences. Only update when explicitly asked.
